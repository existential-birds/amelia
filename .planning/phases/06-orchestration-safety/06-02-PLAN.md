  <behavior>
    ## Concurrency Control (ORCH-01)
    - trigger_fix_cycle(pr_number, profile) acquires per-PR lock before running
    - If lock already held for same PR, sets pending flag and returns (emits PR_FIX_QUEUED event)
    - Pending flag is boolean, not queue -- latest wins since fresh scan covers everything
    - Different PR numbers can hold locks concurrently (no global lock)
    - After fix cycle completes, checks pending flag; if set, runs cooldown then another cycle

    ## Queued Cycles & Cooldown (ORCH-02)
    - After a fix cycle that has a pending follow-up, waits post_push_cooldown_seconds before next cycle
    - Cooldown is interruptible: trigger_fix_cycle sets cooldown event, resetting the timer
    - Timer reset respects max_cooldown_seconds absolute cap
    - When cooldown expires (or cap reached), next cycle starts with fresh scan of all unresolved comments
    - Emits PR_FIX_COOLDOWN_STARTED when cooldown begins, PR_FIX_COOLDOWN_RESET when timer resets

    ## Branch Safety & Divergence Recovery (ORCH-03)
    - Before each fix cycle: fetch origin, checkout head_branch, hard reset to origin/head_branch
    - On divergence (ValueError with "diverged" from safe_push): discard changes, reset to remote, retry fresh
    - Max 2 retries per trigger on divergence (3 total attempts)
    - Emits PR_FIX_DIVERGED on each retry
    - On final failure after retries exhausted: emits PR_FIX_RETRIES_EXHAUSTED event, THEN posts a GitHub PR comment via GitHubPRService (or create_issue_comment) with message "Could not apply fixes -- PR branch changed during fix attempt. Will retry on next cycle." (per locked user decision on divergence notifications)
    - Never operates on main/master -- head_branch comes from PR metadata

    ## Repo-level Git Serialization (Research pitfall 3)
    - A repo-level asyncio.Lock serializes all git operations across PRs sharing the same repo_path
    - Non-git work (classify, develop) runs concurrently across PRs
  </behavior>
  <implementation>
Create `amelia/pipelines/pr_auto_fix/orchestrator.py` with class PRAutoFixOrchestrator:

Constructor takes event_bus (EventBus), github_pr_service (GitHubPRService), and optional config overrides. Maintains:
- `_pr_locks: dict[int, asyncio.Lock]` for per-PR concurrency
- `_pr_pending: dict[int, bool]` for pending flag
- `_cooldown_events: dict[int, asyncio.Event]` for cooldown interruption
- `_repo_locks: dict[str, asyncio.Lock]` for repo-level git serialization

Public method `trigger_fix_cycle(pr_number: int, repo: str, profile: Profile, config: PRAutoFixConfig | None = None) -> None`:
- Checks per-PR lock. If locked, sets pending + sets cooldown event + emits PR_FIX_QUEUED, returns.
- If not locked, acquires lock, runs _run_fix_cycle, then loops on pending with cooldown between.

Private `_run_fix_cycle(pr_number, repo, profile, config)`:
- Acquires repo-level lock for git operations (fetch, reset, checkout)
- Runs the PR auto-fix pipeline (mock-friendly via _execute_pipeline method)
- On divergence, retries up to 2 times with fresh reset
- On final failure (retries exhausted): emits PR_FIX_RETRIES_EXHAUSTED, then calls `_post_final_failure_comment`

Private `_post_final_failure_comment(repo, pr_number)`:
- Uses `self._github_pr_service` to post an issue-level comment on the PR: "Could not apply fixes -- PR branch changed during fix attempt. Will retry on next cycle."
- Use `create_issue_comment` or equivalent method on GitHubPRService (check actual API surface at implementation time)
- Log warning via loguru on failure to post comment (non-fatal -- dashboard event already emitted)

Private `_run_cooldown(pr_number, config)`:
- asyncio.Event-based interruptible wait per research Pattern 2
- Respects max_cooldown_seconds cap

Private `_reset_to_remote(git_ops, branch)`:
- fetch origin branch, checkout branch, reset --hard origin/branch

Private `_emit_event(event_type, pr_number, message, data)`:
- Creates WorkflowEvent and emits via event_bus

Tests mock the pipeline execution (`_execute_pipeline`), GitOperations, and GitHubPRService to test orchestration logic in isolation. Use asyncio fixtures for concurrent trigger testing. Include a test that verifies final divergence failure posts a GitHub PR comment.
